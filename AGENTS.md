# AGENTS.md

## 项目级硬规则

- 每次涉及代码、prompt、配置、交互或数据结构修改前，必须先对照 `docs/vimo-user-prd.md`，确认实现没有偏离 PRD。
- 如果用户当前要求会新增、删除或改变 PRD 中未覆盖的产品行为，必须询问用户是否同步更新 `docs/vimo-user-prd.md`；不要只改代码而让 PRD 落后。
- 不允许在业务代码里通过关键词、短语表、正则或字符串命中判断用户意图。
- 不允许在业务代码里硬编码 AI 回复、确认话术、玩笑边界话术或补救话术。
- 不允许把 prompt 散落在代码、测试外的临时字符串或组件里；运行时 prompt 必须集中维护在 `vimo-go/prompts/`。
- 不允许把用户输入分类为玩笑、日记、想法、待办、备忘、查询、确认、修改、删除等逻辑写成本地规则。
- 不允许为了兜底而让代码冒充 AI 生成回复；模型失败时显示普通错误 UI，不写入 assistant 消息。

## 推荐实现边界

- 用户意图、记录类型、玩笑/不现实边界、是否查询、是否重复/相近、是否接续待确认，都由模型根据系统提示词、当前输入、上下文和历史沉淀记录判断。
- 代码层只做确定性执行：请求转发、JSON 解析、枚举归一化、字段完整性校验、时间格式归一化、数据存取、UI 展示、自动保存条件判断。
- 历史记录查询允许使用语义检索或相似度检索：用用户输入去检索意思相近的日记、待办、想法、备忘，再把候选结果交给模型判断和回复。
- 历史记录查询不允许使用固定关键词路线；检索词应来自用户输入本身或模型生成的查询语义，不应由本地硬编码短语决定。
- 如果需要新增业务能力，优先扩展 `vimo-go/prompts/agent/analyze/` 和结构化 schema，再让代码按模型返回字段执行。

## Prompt 管理

- `vimo-go/internal/agent.LoadSystemPrompt` 是后端加载 Agent prompt 的入口。
- 新增或调整 Agent 行为时，只改 `vimo-go/prompts/agent/analyze/*.md` 中的相关片段。
- 测试中可以使用 stub JSON 和断言字符串，但不得把测试样例变成运行时判断依据。

## 允许的硬编码

- 协议枚举：`intent`、`type`、`status` 等 schema 枚举。
- 纯 UI 文案：按钮、标题、空状态、错误提示、toast、表单标签。
- 纯数据结构映射：类型图标、颜色、tab 名称、路由路径。
- 机械校验：空值、范围、日期格式、JSON 合法性、ID 去重。

## 修改前检查

- 改代码、prompt、配置、交互或数据结构前先读 `docs/vimo-user-prd.md` 的相关章节；如果无法判断相关章节，至少搜索当前需求关键词并确认 PRD 是否覆盖。
- 用户提出新需求或改变既有需求时，最终回复必须说明 PRD 是否同步；如果用户未授权同步 PRD，要明确说明代码已按当前指令处理但 PRD 未更新。
- 改 Agent 流程前先搜索：`rg -n "keyword|关键词|contains|includes|regexp|RegExp|回复|prompt|Prompt" vimo-go vimo-web/src`
- 看到用户原文参与 `contains/includes/regex/switch` 判断时，默认视为违规，除非它只是普通搜索框或数据检索。
- 看到业务代码新增 assistant 固定 `content` 或 `reply` 时，默认视为违规。

## 全局默认补充规则

### 默认开发模式

- 默认直接完成用户要求的开发、修复、重构、排查或文档任务。
- 不主动创建 spec、proposal、design doc 或长方案，除非用户明确要求。
- 修改前先理解现有实现，不基于 wiki 或 docs 直接假设。

### 目标确定与快速通道

- 目标清晰时直接执行，不为了流程而提问。
- 目标不清晰且会影响实现方向时，最多问 3 个关键问题。
- 如果可以安全推断最小合理目标，直接按最小实现执行，并在最终回复中说明假设。
- 用户使用 `快速修`、`直接改`、`不用确认`、`先修`、`hotfix`、`紧急`、`跳过流程` 时，跳过详细目标澄清，按最小安全改动执行。
- 快速通道下除非必须，不新增依赖、不重构、不改公共接口、不扩大范围；涉及数据结构、权限、支付、删除数据、新依赖或大范围架构变化时必须确认。

### Loop 开发闭环

- 开发时遵循：理解上下文 → 修改代码 → 运行最快相关验证 → 根据结果修正 → 必要时更新 wiki → 总结。
- 验证失败时，优先根据错误继续修复并重跑。
- 同一个问题连续失败两次，停止并说明卡点、已尝试方案和需要用户决策的信息。
- 不为了追求闭环而扩大任务范围。

### 项目 Wiki 与 Docs 规则

- 开发前读取 `docs/wiki/index.md`。
- 只读取和任务相关的 wiki/docs，不要一次性读取全部文档。
- 代码、测试和用户当前指令优先于 wiki；wiki 是辅助上下文，不是真理来源。
- 遇到 wiki/docs 和代码不一致时，以代码、测试和用户明确说明为准，并在完成修改时更新过期内容。
- 不把猜测、过期信息或未验证结论写入 wiki；无法确认的内容标记为“待确认”，或者不写入。

### Wiki 初始化规则

- 如果 `docs/wiki/index.md` 不存在，并且当前任务涉及代码修改，自动创建最小 wiki 结构：`docs/wiki/index.md`、`architecture.md`、`conventions.md`、`decisions.md`、`modules/`。
- 初始化时只写已经从代码、配置、测试、README 或用户说明中确认的信息。

### 开发后 Wiki 更新规则

- 修改影响用户可见行为、API、数据结构、配置、架构、测试、命令或重要坑点时，需要更新 `docs/wiki/`。
- 只改相关页面，保持条目短小，并在 `docs/wiki/index.md` 的最近更新中记录一句话摘要。
- 纯格式化、无行为变化的小重命名、临时实验、注释或不影响项目理解的文案微调通常不需要更新 wiki。

### 质量要求

- 保持变更范围小，遵循项目现有风格。
- 不引入新依赖，除非用户明确同意或项目已有模式要求。
- 修改后尽量运行相关测试、lint 或类型检查；如果无法运行，要说明原因。

### 与 CLAUDE.md 保持同步

- `AGENTS.md` 面向 Codex，`CLAUDE.md` 面向 Claude Code。
- 修改项目级开发流程、wiki 规则、快速通道或质量要求时，同步更新两个文件。
