# Vimo / 微默 用户版产品需求说明

## 1. 产品定位

Vimo 是一个面向个人用户的 AI 记录与生活记忆助手。

用户不需要打开复杂的效率工具，也不需要手动选择分类。只要用自然语言说出或输入自己的想法、待办、备忘、经历和情绪，Vimo 就能理解内容，并整理成可查看、可编辑、可回顾的个人记录。

Vimo 的长期方向不是让用户“多写一点”，而是让用户“少操作一点”。用户只需要自然表达，Vimo 帮用户整理生活、任务、想法和记忆。

## 2. 产品愿景

让用户形成一种新的记录习惯：

```text
想到就说
聊到就记
说完就被整理好
以后还能被想起来
```

Vimo 最终希望成为一个越来越懂用户的个人记忆助手。它不仅能记录用户当下说的话，还能在后续对话中理解用户过去经历过什么、关心什么、正在推进什么、最近状态如何。

## 3. 用户痛点

### 3.1 记录成本高

用户每天会产生大量碎片信息：

- 临时想到的待办。
- 想保存的资料。
- 一闪而过的想法。
- 当天发生的事情。
- 情绪和复盘。

但现有工具通常要求用户打开 App、打字、选择分类、设置时间、保存。步骤越多，用户越容易放弃记录。

### 3.2 记录之后缺少整理

用户即使记录了内容，也经常面临：

- 不知道放在哪。
- 事后找不到。
- 待办没有提醒。
- 日记没有结构。
- 想法和备忘混在一起。

记录本身没有形成可持续价值。

### 3.3 用户不想专门写日记

很多用户不是没有日记价值的内容，而是懒得专门写日记。

用户可能已经在和 AI 聊自己的生活、工作、情绪和想法，但这些对话通常聊完就结束了，没有沉淀成可回顾的日记和记忆。

### 3.4 AI 不够懂用户

普通 AI 对话缺少长期记忆。用户每次重新打开对话时，都像重新认识 AI。

用户希望 AI 能知道：

- 我最近在忙什么。
- 我之前为什么焦虑。
- 我上次提到的项目是什么。
- 我最近有哪些反复出现的问题。
- 我曾经有哪些想法和计划。

## 4. 目标用户

### 4.1 核心用户

- 经常有碎片待办和想法的人。
- 习惯使用 AI 对话的人。
- 想记录生活但坚持不了写日记的人。
- 容易忘事、需要提醒的人。
- 有复盘、情绪记录、个人成长需求的人。

### 4.2 典型用户画像

#### 用户 A：忙碌的职场人

经常临时想到任务和安排，但没时间打开复杂工具整理。希望能一句话记下，并在合适时间被提醒。

#### 用户 B：喜欢和 AI 聊天的人

会和 AI 聊工作压力、生活状态和想法，但希望这些聊天能沉淀成日记和长期记忆。

#### 用户 C：想记录生活但坚持不了的人

不想每天专门写日记，但希望回头能看到自己每天发生了什么、想过什么、情绪如何变化。

## 5. 核心价值主张

### 5.1 对用户的承诺

Vimo 帮你把自然表达变成有用记录。

你不需要先想清楚这是待办、日记、备忘还是想法。你只需要说出来或打出来，Vimo 帮你理解、分类、整理，并在需要时提醒你。

### 5.2 与普通记录工具的区别

普通记录工具要求用户主动整理。

Vimo 的核心是：

```text
用户表达
-> Vimo 理解
-> 自动整理
-> 用户确认
-> 后续可回顾、可检索、可被对话引用
```

### 5.3 与普通 AI 聊天的区别

普通 AI 聊天主要解决当下问题。

Vimo 的 AI 对话会沉淀长期价值：

- 有价值的对话可以变成日记。
- 用户确认后的日记会进入个人记忆库。
- 后续对话可以按日期、关键词或语义引用相关记忆。
- Vimo 会逐渐更理解用户。

## 6. 产品核心闭环

### 6.1 MVP 记录闭环

```text
自然语言输入
-> AI 理解内容
-> 自动分类
-> 生成结构化记录
-> 用户确认、编辑或删除
-> 保存到对应列表
```

### 6.2 待办提醒闭环

```text
用户说出任务
-> Vimo 判断为待办
-> 识别时间
-> 有明确时间则生成提醒计划
-> 时间不明确则进入待确认
-> 用户补充后保存
-> 到点后 Vimo 主动提醒
-> 用户完成、推迟、修改或删除
```

这个闭环需要同时满足两件事：

- 用户能清楚知道自己还有什么没做、什么已经做了、做过什么、接下来打算什么时候做。
- Vimo 能清楚知道自己什么时候需要主动提醒用户、哪些提醒已经执行、哪些还需要继续跟进。

### 6.3 聊天生成日记闭环

```text
用户和 Vimo 闲聊
-> Vimo 判断是否有日记价值
-> 生成第一人称日记草稿
-> 用户确认、修改或忽略
-> 保存到日记库
```

### 6.4 记忆反哺对话闭环

```text
用户再次对话
-> Vimo 根据日期、关键词或语义检索日记库
-> 带入少量相关记忆
-> 给出更理解用户上下文的回复
```

### 6.5 主动提醒与主动关怀闭环

Vimo 不应只是被动等待用户打开，也可以在用户明确需要时主动出现。

```text
用户设置待办或回访
-> Vimo 在约定时间主动发起提醒
-> 用户完成、稍后提醒、调整时间或继续聊
-> Vimo 根据用户反馈更新记录
```

主动交互的核心原则是：

```text
主动
克制
可控
有边界
```

任务到点提醒属于确定性能力，应优先进入早期版本。情绪和状态回访更接近陪伴能力，必须基于用户明确表达或明确授权，避免让用户感到被打扰或被监视。

### 6.6 任务履约闭环

用户设置待办后，Vimo 不只是保存一条记录，还应承担后续履约责任。

```text
用户待办：明天下午三点交周报
Vimo 履约：明天下午三点主动提醒用户交周报
```

用户看到的是自己的任务状态：

```text
未完成
已完成
已提醒
已推迟
已取消
```

Vimo 需要根据用户的操作更新任务状态：

```text
用户点击已完成 -> 待办进入已完成
用户选择稍后提醒 -> 生成新的提醒计划
用户修改时间 -> 更新提醒计划
用户删除待办 -> 取消对应提醒
```

这个机制的价值是让 Vimo 从“记录工具”变成“会履约的助手”。用户不仅知道自己记录过什么，也知道哪些事情还没做、哪些已经做了、哪些正在等待未来某个时间处理。

## 7. 核心场景

### 7.1 快速记录待办

用户说：

```text
明天下午三点提醒我交周报。
```

Vimo 应理解为待办，并记录提醒时间。

用户看到：

```text
已记录，明天下午三点提醒你交周报。
```

价值：

- 用户不用打开待办工具。
- 用户不用手动设置提醒。
- 用户不用选择分类。

### 7.2 记录没有明确时间的待办

用户说：

```text
这周找时间整理一下电脑桌面。
```

Vimo 应理解为待办，但不强行设置提醒。

该内容进入待确认，用户稍后可以补充时间或确认是否需要提醒。

价值：

- 不打断用户当前记录。
- 避免 AI 错误设置提醒。
- 让模糊信息也能被接住。

### 7.3 记录日记

用户说：

```text
今天和朋友吃饭挺开心，聊了很多以前的事情。
```

Vimo 应识别为日记，并保存到对应日期。

价值：

- 用户无需打开日记应用。
- 用户无需组织完整文字。
- 简短表达也能沉淀为生活记录。

### 7.4 闲聊自动生成日记

用户和 Vimo 对话：

```text
用户：今天有点累，项目推进不太顺，感觉一直卡在任务拆解上。
Vimo：听起来你今天压力挺大，也许可以先把最卡的点拆小一点。
```

Vimo 可以生成日记草稿：

```text
今天项目推进不太顺，我有点累，也有一些压力。主要卡在任务拆解和执行节奏上。
```

注意：

- 日记正文使用用户第一人称。
- Vimo 的回复不直接混入日记正文。
- Vimo 的回复可以作为单独的对话记录或建议摘要保存。
- 用户必须可以确认、修改或忽略这篇日记。

价值：

- 用户不用专门写日记。
- 聊天内容不再聊完就丢。
- Vimo 可以逐渐积累用户的生活上下文。

### 7.5 记录备忘

用户说：

```text
公司 Wi-Fi 密码是 xxxx。
```

Vimo 应识别为备忘，并保存到备忘列表。

价值：

- 用户可以快速保存重要信息。
- 后续可以通过关键词查找。

### 7.6 记录想法

用户说：

```text
我想到一个产品点子，可以做一个语音输入后自动变成任务卡片的工具。
```

Vimo 应识别为想法，并保存到想法列表。

价值：

- 灵感不会丢。
- 想法和待办、备忘、日记分开保存。

### 7.7 通过日期或关键词回忆过去

用户问：

```text
我最近是不是经常因为项目拆解焦虑？
```

Vimo 可以检索用户确认保存过的日记，结合相关日期和主题回复。

价值：

- 用户可以回顾自己的状态变化。
- Vimo 能基于真实记忆对话。
- 对话更连续，更像一个长期陪伴助手。

### 7.8 到点主动提醒待办

用户说：

```text
明天下午三点提醒我交周报。
```

到明天下午三点时，Vimo 主动发起一条提醒：

```text
到时间了，该交周报了。现在要处理吗？
```

用户可以选择：

- 已完成。
- 稍后提醒。
- 修改时间。
- 删除待办。
- 和 Vimo 继续聊一下这个任务。

价值：

- 待办不是只被保存，而是真的在关键时间被想起来。
- 提醒以对话形式出现，比普通通知更自然。
- 用户可以直接在提醒里完成下一步操作。

### 7.9 主动情绪回访

用户说：

```text
我现在心情不太好，想冷静一小时。
```

Vimo 可以理解为一次“状态回访”请求，并回复：

```text
好，我一小时后再来问问你感觉怎么样。
```

一小时后，Vimo 主动发起：

```text
刚才你说想冷静一小时，现在感觉好一点了吗？
```

用户可以选择：

- 好多了。
- 还是不太好。
- 想聊聊。
- 不用再提醒。

价值：

- Vimo 不只是工具，也像一个有温度的陪伴者。
- 用户在低落或焦虑时，可以获得轻量但及时的回访。
- 回访结果可以沉淀为日记或情绪记录，但必须经过用户确认。

## 8. 功能需求

### 8.1 自然语言记录入口

用户可以通过输入或语音表达内容。

MVP 阶段优先支持文本输入，后续支持语音输入。

需求：

- 用户可以快速输入一段自然语言。
- 输入后 Vimo 自动理解内容。
- 用户不需要手动选择分类。
- Vimo 需要给出简洁反馈。

### 8.2 AI 意图栈与自动分类

Vimo 需要先识别用户本轮输入的主意图，再识别可能存在的副意图，并把可沉淀内容整理成候选记录。

意图栈包含：

- 主意图：决定本轮主要回复和主要执行动作。
- 副意图：保留附带但重要的信息，例如情绪信号、日记候选、想法候选、备忘候选或上下文更新。
- 候选记录：每个可沉淀内容都可以形成独立候选，绑定对应意图。
- 执行计划：判断候选是自动执行、预览、待确认、追问还是只回复。
- 意图追踪：记录本轮是否接续未收口上下文、主要风险原因、确认门原因、被放弃的替代解释和状态迁移，用于调试和回归，不面向用户展示完整推理。
- 未收口上下文：待确认任务需要保留 `intent`、`record_action`、`target_id`、`related_ids` 和执行计划，下一轮“是的/不要/改一下”等短回复才能接回原任务。
- 本地上下文缓存：Demo 阶段未接数据库时，对话消息、未收口上下文、待确认候选和当前打开的待确认项保存在浏览器本地缓存中，刷新页面后仍可继续接续；用户可以手动删除不需要的本地上下文，避免未收口项堆积。

候选记录类型包括：

- 待办。
- 日记。
- 备忘。
- 想法。
- 普通闲聊。
- 待确认。

分类规则：

- 有行动、计划、提醒意图时，优先识别为待办。
- 描述经历、情绪、复盘时，识别为日记或日记候选。
- 保存信息、密码、地址、资料时，识别为备忘。
- 灵感、创意、产品点子时，识别为想法。
- 不确定时进入待确认，不强行归类。
- 普通问答或闲聊不应默认保存。
- 副意图默认不喧宾夺主；日记、情绪、长期记忆和主动回访类副意图保存前需要确认。明确、字段完整且高置信的待办、备忘或想法副候选可以自动执行。
- 日记、情绪、长期记忆和主动回访类副意图必须更保守，保存前需要用户确认。
- 目标不唯一或目标低置信的修改/删除、模糊提醒时间、隐私日记、长期记忆和主动回访属于 hard stop 场景；即使 AI 置信度高，也必须经过确认门或追问。
- 删除已保存记录时，如果 AI 给出唯一 `target_id` 且目标置信度高，可以直接软删除进入回收站；目标不明确时必须确认。
- 多动作输入不应靠本地关键词或针对性 prompt 硬规则处理，应由 AI 拆成多条候选任务和执行计划，代码层按结构化任务依次执行、预览或确认。
- Vimo 的回复需要自然承接用户本轮语境。行动和情绪同时出现时，应轻量回应情绪，再明确确认主动作，不应只机械回复“已记录/已提醒”。

### 8.3 待办记录

待办记录需要支持：

- 标题。
- 原始内容。
- 时间信息。
- 提醒状态。
- 完成状态。
- 履约状态。
- 编辑。
- 删除。

当时间明确时，Vimo 应创建提醒。

当时间不明确时，Vimo 应进入待确认。

用户需要能清楚区分：

- 还没做的待办。
- 已经完成的待办。
- 已经过期但未完成的待办。
- 已提醒但用户选择稍后处理的待办。
- 已取消或删除的待办。

当用户修改、完成或删除待办时，对应的提醒计划也需要同步变化。

### 8.4 时间识别与提醒

Vimo 需要识别用户表达中的时间。

需求：

- 明确时间可以直接生成提醒。
- 只有日期没有具体时间时，可以使用默认提醒时间。
- 模糊时间不强行提醒。
- 用户可以修改提醒时间。
- 用户可以关闭或开启通知权限。

### 8.5 待确认

待确认用于承接 AI 暂时无法确定的内容。

待确认需要有清晰状态：

- `open`：未收口但不只是在等单个字段。
- `waiting_field`：正在等待用户补充字段，例如提醒时间、目标记录或是否提醒。
- `ready_to_execute`：信息已补齐，等待代码层执行或展示。
- `executed`：已经保存、更新或删除。
- `dismissed`：用户忽略或删除。

进入待确认的情况：

- 待办缺少明确时间。
- 时间表达模糊。
- 分类不确定。
- 日记日期不确定。
- 内容可能需要用户补充信息。

需求：

- 用户可以查看待确认内容。
- 用户可以补充时间。
- 用户可以修改分类。
- 用户可以确认保存。
- 用户可以删除或忽略。
- 待确认不应打断用户当前表达。

### 8.6 日记

日记来源包括：

- 用户主动记录。
- 用户与 Vimo 的对话中产生的日记候选。

日记需求：

- 按日期保存。
- 支持标题。
- 支持正文。
- 支持关键词或主题。
- 支持编辑和删除。
- 支持查看来源对话。

日记正文要求：

- 使用用户第一人称。
- 不加入用户没有表达过的事实。
- 不把 Vimo 的回复混入用户第一人称正文。

### 8.7 闲聊生成日记草稿

Vimo 可以在用户聊天后判断是否需要生成日记草稿。

适合生成日记的内容：

- 用户讲述当天经历。
- 用户表达明显情绪。
- 用户复盘工作、学习、关系或生活事件。
- 用户提到值得回顾的决定、想法或变化。

不适合生成日记的内容：

- 单纯知识问答。
- 临时查询。
- 无个人上下文的闲聊。
- 明显敏感且用户未表达保存意愿的内容。

要求：

- 默认生成草稿，不默认自动保存。
- 用户可以确认、修改、忽略。
- 用户可以关闭该能力。
- 草稿需要说明来自哪段对话。

### 8.8 日记库检索增强对话

Vimo 可以在用户后续对话中检索已保存的日记。

检索方式：

- 日期检索，例如“昨天”“上周五”“7 月 1 日”。
- 关键词检索，例如“项目”“朋友”“健身”“焦虑”。
- 语义检索，例如“最近让我压力大的事情”。

要求：

- 只使用用户确认保存的内容。
- 只带入与当前问题相关的少量记忆。
- 不大段暴露用户历史原文。
- 用户可以查看引用来源。
- 用户可以删除不希望继续使用的记忆。

### 8.9 备忘

备忘用于保存重要信息。

需求：

- 支持标题。
- 支持原始内容。
- 支持关键词。
- 支持编辑和删除。
- 后续支持搜索。

### 8.10 想法

想法用于保存灵感和创意。

需求：

- 支持标题。
- 支持正文。
- 支持创建时间。
- 支持编辑和删除。
- 后续可扩展为任务或项目。

### 8.11 识别结果反馈

用户提交内容后，Vimo 需要给出简洁反馈。

反馈要求：

- Vimo 支持用户选择 AI 回复风格倾向。当前内置 `INTJ`、`ENFJ`、`ISTP`、`ENFP` 和 `custom`；这些值只影响回复组织方式和语气，不用于判断用户人格，也不改变意图识别、保存、删除或提醒执行规则。
- 用户提交后，Vimo 应先启动快路：快路模型生成一句即时承接，并判断本轮是否需要继续进入慢路。
- 快路的 `text/route` 必须通过模型结构化 JSON 模式一次性返回；后端只把解析后的 `text` 发给前端做逐字渲染，不允许把协议 JSON 作为用户可见流式文本输出。
- 慢路的完整识别结果也必须通过模型结构化 JSON 模式返回；非 JSON 或缺少必要回复字段时，应展示普通错误 UI，不写入 assistant 兜底话术。
- 纯寒暄或普通闲聊可以只走快路直接回复，不进入慢路，不生成意图栈、记录卡片或待确认项。
- 涉及记录、提醒、日记、想法、备忘、查询历史、修改、删除、配置调整、未收口上下文接续或任何不确定是否只是闲聊的输入，必须继续进入慢路；慢路负责完整识别和真实执行准备。
- 快路反馈不能提前声称已经保存、已经删除、已经修改完成或已经设置提醒成功。
- 慢路应接收快路已展示的承接文本上下文，并像同一个助手在续写，不重复快路已经说过的寒暄、称呼或情绪承接。
- 如果慢路最终回复包含快路已经展示的开头文本，用户界面应只展示慢路新增部分，避免同一轮回复重复。
- 快路和慢路都应接收最近可见聊天消息，用于理解短追问、代词、上一句闲聊语境和重复提问；普通闲聊追问不应因为存在未收口记录而被强行识别成日记或待确认，重复问同一问题时应体现“已经聊过”的连续感，而不是每次像第一次回答。
- 慢路继续负责完整意图栈、候选记录、风险门控和执行计划，只有慢路完成后才展示最终识别结果、记录卡片和待确认项。
- 慢路判断为明确可自动执行的记录动作时，后端应负责执行保存、更新、删除或恢复，并通过事件把执行结果返回给前端；需要用户确认的动作仍进入待确认。
- Vimo 可以在 AI 消息内展示轻量处理进度；进度必须来自后端真实代码事件，例如快路开始、慢路分析、模型返回、候选生成和执行计划，不允许由模型编造完成状态。
- 快路和慢路的处理过程应各自用轻量折叠摘要展示，摘要显示“处理中/已处理”和耗时；处理完成后默认折叠，用户可展开查看过程细节。
- 用户开启思考模式时，provider reasoning 可以作为本轮临时过程信息展示，但默认不写入长期记录或本地聊天历史持久化。
- AI 回复需要流式渲染，优先按模型输出增量展示；前端视觉层应保证文字逐字出现，而不是整段一次性冒出。
- 对话区的思考中状态在快路文字输出期间保持展示；如果本轮 `route=chat_only`，快路完成后立即消失；如果本轮进入慢路，则直到慢路最终回复开始输出首字时再消失。
- 识别明确时，直接展示结果。
- 识别不明确时，提示用户稍后确认。
- 不要求用户每次都进行复杂操作。
- 用户始终可以修改 AI 的结果。

### 8.12 主动交互机制

Vimo 需要支持在合适时间主动向用户发起交互。

主动交互分为两类：

#### 到点提醒

适用场景：

- 明确时间的待办。
- 用户主动设置的提醒。
- 用户修改后的提醒时间。

要求：

- 到点后 Vimo 主动提醒用户。
- 提醒内容应简洁明确。
- 用户可以直接完成、稍后提醒、修改时间或删除。
- 提醒需要尊重通知权限和免打扰设置。
- 用户完成或删除待办后，后续提醒应自动停止。
- 用户选择稍后提醒时，应形成新的提醒计划。
- 用户修改时间时，应同步更新提醒计划。

#### 履约状态

每个需要 Vimo 主动执行的提醒，都应有清晰状态。

面向用户可以表达为：

- 等待提醒。
- 已提醒。
- 已完成。
- 已推迟。
- 已取消。
- 提醒失败。

用户不需要理解内部执行细节，但需要能知道为什么 Vimo 会在某个时间主动出现，以及这次提醒来自哪条待办或回访请求。

#### 主动回访

适用场景：

- 用户明确要求稍后回访，例如“一个小时后问我怎么样”。
- 用户表达需要冷静、休息、稍后再处理。
- 用户主动要求 Vimo 稍后陪他继续聊。

要求：

- 不默认对所有情绪内容发起回访。
- 只有用户明确表达或明确授权后才回访。
- 回访语气要轻量、温和，不做诊断。
- 用户可以关闭主动回访。
- 用户可以设置不打扰时间。
- 回访内容如需写入日记，必须由用户确认。

### 9.2 待办页

用于管理用户的任务。

需求：

- 查看未完成待办。

## 10. 隐私与信任原则

Vimo 涉及个人日记、情绪、任务和生活记忆，因此必须优先保护用户信任。

原则：

- 不偷偷保存用户不想保存的内容。
- 闲聊生成日记默认先生成草稿。
- 用户确认后才进入日记库。
- 用户可以编辑、忽略和删除。
- 用户可以关闭聊天生成日记。
- 用户可以关闭记忆检索。
- 用户可以关闭主动回访。
- 用户可以设置免打扰时间。
- 情绪回访必须基于用户明确表达或授权。
- 用户可以查看记忆引用来源。
- AI 不应编造用户没有表达过的事实。

## 13. 关键风险

### 13.1 AI 分类错误

风险：

Vimo 可能把内容分错类，影响用户信任。

应对：

- 允许用户修改分类。
- 低置信度进入待确认。
- 保留原始内容。

### 13.2 时间识别错误

风险：

Vimo 可能误解“明天”“下周五”“这周找时间”等表达。

应对：

- 明确时间才直接提醒。
- 模糊时间进入待确认。
- 用户可以修改提醒时间。
- 回归测试需要覆盖模糊时间不能误自动提醒。

### 13.3 用户感觉被偷偷记录

风险：

闲聊自动写日记如果处理不好，会让用户觉得隐私被侵犯。

应对：

- 默认生成草稿，不默认保存。
- 明确展示来源对话。
- 支持关闭功能。
- 支持删除所有相关记录。
- 日记、长期记忆和主动回访进入 hard stop gate，保存或启用前必须确认。

### 13.4 AI 编造日记

风险：

AI 可能补充用户没有说过的事实或过度解读情绪。

应对：

- 日记只基于用户表达。
- Vimo 回复和用户日记分开。
- 保存前用户确认。
- 支持编辑和删除。

### 13.5 主动交互打扰用户

风险：

如果 Vimo 频繁主动发起聊天，用户会觉得被打扰，甚至产生被监视感。

应对：

- 到点提醒只针对明确设置的待办或提醒。
- 情绪回访必须由用户明确触发或授权。
- 提供主动提醒和主动回访开关。
- 提供免打扰时间。
- 回访频率需要克制，不连续追问。
- 主动回访第一阶段只作为高风险意图进入确认门；未完成真实主动回访调度能力前，不应静默启用。

### 13.6 履约状态不一致

风险：

用户修改或删除待办后，如果 Vimo 仍然在原时间提醒，会严重损害信任。

应对：

- 用户修改时间后，提醒计划必须同步更新。
- 用户完成或删除待办后，后续提醒必须停止。
- 提醒失败或错过时，需要有可理解的状态提示。
- 待办页需要展示最新状态，避免用户不知道任务到底处理到哪一步。

### 13.7 情绪陪伴越界

风险：

Vimo 的情绪回访可能被用户理解为心理咨询或专业建议。

应对：

- 回访只做轻量关心，不做诊断。
- 不替用户判断心理状态。
- 对严重负面表达，优先建议用户联系可信的人或专业帮助。
- 不把敏感情绪内容自动写入日记。

## 14. 产品结论

Vimo 的核心不是做一个更复杂的效率系统，而是做一个更低成本的个人记录和记忆入口。

第一阶段，Vimo 需要先跑通：

```text
自然语言输入
-> AI 理解
-> 自动分类
-> 用户确认
-> 保存记录
```

第二阶段，Vimo 应验证：

```text
自然聊天
-> 自动生成日记草稿
-> 用户确认保存
```

第三阶段，Vimo 才进入更有差异化的能力：

```text
日记和记录成为用户记忆库
-> 后续对话能引用这些记忆
-> Vimo 越来越懂用户
```

同时，Vimo 应逐步建立主动交互能力：

```text
到点主动提醒任务
-> 用户授权后主动回访状态
-> 在克制和可控的前提下提供陪伴感
```

一句话总结：

```text
Vimo 不是让用户更努力地记录，而是让用户在自然表达中自动留下有价值的生活记忆。
```
